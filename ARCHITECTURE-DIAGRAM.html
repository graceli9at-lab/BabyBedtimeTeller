Data Flow
What happens when you click "Generate"
1
User submits the form
React collects name, age, characters, length. The submit handler calls generate(formData) in the useStoryStream hook, which fires a fetch() POST to /api/story with an AbortController attached.
Client
2
Server validates input
Zod schema checks all four fields — types, lengths, allowed values. HTML characters are stripped. If validation fails, a 400 is returned immediately and Claude is never called. Rate limiter checks 10 req/min per IP.
Middleware
3
SSE headers open the stream
Server sets Content-Type: text/event-stream, Cache-Control: no-cache, calls res.flushHeaders(). This immediately opens the connection so the client can start reading — even before the first word arrives from Claude.
Performance
4
Claude API called with streaming
claude.messages.stream() opens a persistent HTTPS connection to Anthropic. The system prompt sets age-appropriate tone and target length (80–250 words). The user prompt injects name and characters as labelled data — not as instructions — to prevent prompt injection.
AI · Security
5
Each token streamed to browser
Every text_delta event from Claude is immediately forwarded as data: {"type":"chunk","text":"…"}\n\n. The client's ReadableStream reader receives it within milliseconds. First word typically appears in ~800ms — well under the 30s target.
Streaming · Performance
6
React renders each chunk
setStoryText(prev => prev + chunk) appends each token to state. React re-renders only the changed text node. The blinking cursor signals the story is still arriving. Paragraphs are split on \n\n for display.
Client · UX
7
Telemetry written on completion
Once the stream closes, the server appends one JSON line to data/telemetry.jsonl: request ID, child age, story length, time_to_first_token_ms, total_generation_ms, token counts, and status. No name, no characters text, no story content — zero PII.
Observability · Privacy

Design Decisions
Key tradeoffs

SSE streaming vs. wait-and-respond

✦

Streaming: First word in ~800ms, feels alive, tolerates slow networks gracefully, lower perceived latency even if total time is the same.

○

Wait-and-respond: Simpler code, easier error handling, but 8–15s of blank screen — fatal for a product where first impressions matter.

✓ Chosen: SSE streaming

Express server vs. frontend-only

✦

Backend proxy: API key never reaches the browser. CORS locked to localhost. Rate limiting and validation happen server-side where they can't be bypassed.

○

Frontend-only: Simpler setup, but API key in client code — visible in DevTools, leakable if ever deployed or shared.

✓ Chosen: Express server (security requirement)

JSON file vs. SQLite for telemetry

✦

JSONL append-log: Zero native dependencies, no compilation, works reliably across WSL + Windows. Queryable with Node.js built-ins.

○

SQLite: Proper indexing, better for large-scale queries. But better-sqlite3 requires native compilation — fragile in WSL environments.

✓ Chosen: JSONL (upgrade path to SQLite is clean)

fetch() + ReadableStream vs. EventSource

✦

fetch + ReadableStream: Supports POST with JSON body, full control over headers, can attach AbortController to cancel mid-stream.

○

EventSource API: Built for SSE but only supports GET requests — would require a separate session-ID handshake to pass form data.

✓ Chosen: fetch + ReadableStream

Stateless sessions vs. user accounts

✦

Stateless: No auth, no database, no GDPR surface. Simpler code, faster to build, no session management bugs.

○

User accounts: Story history, personalization over time — but adds significant complexity and a data responsibility.

✓ Chosen: Stateless (explicit MVP scope)

Mock mode vs. always-live API

✦

Mock mode: Full UI testing without an API key or cost. Realistic word-by-word delay simulates streaming. Toggled by one env var.

○

Always-live: Tests the real integration path but adds cost per test run and blocks development without a key.

✓ Chosen: Mock mode default (MOCK_MODE=true)

Technical Insights
Patterns worth noting

Prompt injection is mitigated by data labelling, not filtering

User input (name, characters) is injected into the prompt as explicitly labelled data: "The child's name is {name}. Include: {characters}." — not concatenated into instructions. This means even if a user types "ignore previous instructions", it reads as a character name to Claude, not a directive. The system prompt is instruction-only; user content is confined to the user turn.

AbortController prevents ghost streams

If a parent submits the form a second time before the first story finishes, abortRef.current.abort() cancels the in-flight fetch. Without this, you'd have two SSE readers racing to write to the same React state — producing garbled, interleaved story text. The abort signal also propagates server-side, closing the Claude stream early and saving tokens.

res.flushHeaders() is critical for streaming to work

Without calling res.flushHeaders() before the async Claude call begins, Express buffers the response until the first res.write(). With it, the browser receives the text/event-stream headers immediately — establishing the SSE connection — so the very first token can be delivered with zero extra round-trips.

Telemetry is deliberately non-blocking and non-critical

The logRequest() call is wrapped in a try/catch that silently swallows errors. If the disk is full, the data directory is missing, or the JSON serialization fails — the story still delivers. Telemetry is observability infrastructure, not product logic. Failing silently here is the correct design; a crash would be a bug.

token max_tokens caps double as UX and cost controls

Setting max_tokens: 350 / 600 / 900 for short/medium/long serves two purposes: (1) it bounds story length so the reading time matches the label, and (2) it caps the per-request cost to ~$0.002–$0.005. Without this, a pathological prompt could generate thousands of tokens — running up cost and making the "long" option 10× longer than expected.

Risk Management
Risk register
Risk Likelihood Impact Mitigation
API key exposed in browser
 Low High Key lives in .env server-side only. .gitignore excludes it. CORS blocks cross-origin requests. Browser never receives the key — by design.
Prompt injection via user input
 Med Med User input is labelled data in the user turn, not part of the system prompt. HTML characters stripped server-side. Max length 200 chars limits injection surface. Claude's instruction-following makes data-context injection low-risk.
Runaway API cost
 Low Med Hard max_tokens cap per story. Rate limiter: 10 req/min per IP. Mock mode on by default — no cost during development. Set a spend cap in Anthropic console as additional backstop.
Claude API downtime / timeout
 Low Med Server catches API errors and emits a safe {"type":"error"} SSE event. Client shows a friendly error state with a retry button. Mock mode available as instant fallback for demos.
Inappropriate story content
 Low High System prompt explicitly constrains tone: "soothing", "calming", "ends with child falling asleep". Claude's built-in safety filters apply. Character input is max 200 chars and clearly labelled as creative input, not instructions.
PII stored in telemetry
 Low Med Telemetry schema stores only: age (integer), story length (enum), timing, token counts, status. Child name, character names, and story text are deliberately excluded. No user identity. File is gitignored.
WSL/Windows filesystem git conflicts
 High Low Known WSL issue: git cannot chmod files on /mnt/c/. Mitigated by running all git operations from the Linux home directory (~/), syncing files with rsync before committing. No impact on running the app.
Stale telemetry file (unbounded growth)
 Med Low Each record is ~200 bytes. At 100 stories/day, the file grows ~7MB/year — trivial locally. Metrics query filters to last 7 days. Manual rotation (rename file) is the upgrade path if needed.
